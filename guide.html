01Overview
Chapter One

概要

このサンプルが示すこと、なぜ TypeScript × Python なのか。

このリポジトリは、TypeScript + Python の組み合わせで AI 駆動開発を行う場合の CI の在り方を示すサンプルです。 シンプルな住所入力フォーム（入力画面 → 完了画面の 2 画面）を題材にしています。

このサンプルで確認できること

  • フロントエンド：Zod によるスキーマ定義、react-hook-form による状態管理、Vitest による純粋関数テスト
  • バックエンド：FastAPI + Pydantic v2、uv によるパッケージ管理、pytest による純粋関数テスト
  • CI：GitHub Actions で backend / frontend / e2e の 3 ジョブを並列実行
  • E2E：Playwright で画面操作を自動化（PR 時にトリガー）
ユーザー操作
住所入力 → 送信ボタン → 完了画面
フロント
バリデーション
Zod safeParse → NG → エラー表示 OK → POST /api/validate
バックエンド
Pydantic 型チェック → validate_address() → NG → { valid: false, errors } OK → { valid: true, address }

ディレクトリ構成

frontend / backend / e2e の 3 領域に分かれたシンプルな構成。

ci-sample/ ├── .github/workflows/ │ └── ci.yml ← GitHub Actions（3 ジョブ並列） │ ├── backend/ │ ├── pyproject.toml ← uv 依存関係定義（Python 3.13） │ ├── uv.lock │ └── app/ │ ├── main.py ← FastAPI アプリ・CORS 設定 │ ├── models.py ← Pydantic モデル定義 │ ├── domain/ │ │ └── validation.py ← 純粋関数（テスト対象） │ ├── api/routes/ │ │ └── validate.py ← HTTP ルーター │ └── tests/ │ └── test_validation.py │ ├── frontend/ │ ├── package.json ← Vite 6 / Vitest 3 / Biome │ ├── vite.config.ts │ ├── e2e/ │ │ └── address-form.spec.ts ← Playwright E2E │ └── src/features/address/ │ ├── validation.ts ← 純粋関数（テスト対象） │ ├── validation.test.ts │ ├── AddressForm.tsx │ ├── FormField.tsx │ ├── CompletionScreen.tsx │ └── api.ts │ └── guide.html ← このドキュメント

ハイライト（validation.py / validation.ts） のように、UI・HTTP 層から分離した純粋関数として定義することで、フレームワーク依存なしにテストできます。

03Pipeline
Chapter Three

CI パイプライン

3 ジョブが並列で走り、全部グリーンになってはじめてマージできる。

GitHub Actions で backend・frontend・e2e の 3 ジョブを並列実行します。 それぞれ独立しているため、どれか 1 つが失敗しても残りは続行され、問題箇所がすぐ特定できます。

3 ジョブとも PR 時のみトリガーします。マージ前に問題を検知するのが目的で、push（main）での再実行は不要と判断しています。

Job 1
backend
uv sync --frozen
ruff check（lint）
ruff format --check
mypy（型チェック）
pytest（ユニットテスト）
Python 3.13 · uv 管理
Job 2
frontend
npm ci
tsc --noEmit（型チェック）
biome check（lint）
vite build
vitest run（ユニットテスト）
Node.js 22 · Vite 6 · Vitest 3
Job 3
e2e
uvicorn 起動（BE）
vite preview 起動（FE）
wait-on で疎通確認
playwright test
Chromium · PR 時のみ

主要な設定ポイント

  • astral-sh/setup-uv@v8.2.0 で uv をセットアップ（uv sync --frozen でロックファイル通りにインストール）
  • actions/cache@v5 で node_modules と uv の仮想環境をキャッシュ
  • E2E はバックエンドを nohup uvicorn、フロントを nohup vite preview でバックグラウンド起動し、npx wait-on で疎通を待つ
  • 全アクションを v5 に統一（Node.js 20 非推奨警告を回避）
# .github/workflows/ci.yml の骨格
on:
  push:
    branches: [main]
  pull_request:

jobs:
  backend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
      - uses: astral-sh/setup-uv@v8.2.0
      - run: uv sync --frozen
      - run: uv run ruff check .
      - run: uv run mypy app
      - run: uv run pytest

  frontend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v5
      - uses: actions/setup-node@v5
        with: { node-version: "22" }
      - run: npm ci && npx tsc --noEmit && npm run build && npx vitest run

  e2e:
    runs-on: ubuntu-latest
    steps:
      - # BE と FE をバックグラウンドで起動して Playwright を実行
      - run: npx playwright test
04Testing
Chapter Four

テスト設計

VB.net との比較で見る「テストしやすいコード」の原則。

VB.net プロジェクトでよく見られる課題は、ビジネスロジックが UI や Excel COM に直接依存していることです。 この構造ではロジック単体でのテストが書けず、「動かして目で確認する」しかなくなります。

観点 VB.net（従来型） このサンプル（TypeScript × Python）
ロジックの場所 Form クラスや Excel シート関数に直接記述 validation.ts / validation.py に分離
テストの方法 画面を操作して目視確認（自動化困難） 純粋関数を直接 call してアサート
依存 UI フレームワーク・Excel COM・DB が必要 フレームワーク依存ゼロ。入力 → 出力のみ
AI との相性 副作用だらけで AI が生成しにくい 純粋関数なので AI が生成・修正しやすい
原則： 「テストできない」ではなく「テストしやすい設計になっていない」。 ロジックを純粋関数として切り出すだけで、言語を問わず自動テストが書けるようになります。

Python 側のテスト例

# tests/test_validation.py
import pytest
from app.domain.validation import validate_address
from app.models import AddressRequest

# ロジック単体テスト — FastAPI も DB も起動不要
def test_valid_address():
    req = AddressRequest(
        postal_code="123-4567", prefecture="東京都",
        city="渋谷区", street_address="1-2-3"
    )
    assert validate_address(req) == []

@pytest.mark.parametrize("code", ["1234567", "abc-defg", ""])
def test_invalid_postal_code(code):
    req = AddressRequest(postal_code=code, prefecture="東京都",
                         city="渋谷区", street_address="1-2-3")
    errors = validate_address(req)
    assert any(e.field == "postal_code" for e in errors)

TypeScript 側のテスト例

// validation.test.ts
import { z } from "zod"
import { addressSchema } from "./validation"

// React も Vite も起動不要 — スキーマを直接 safeParse
test("正しい住所はパスする", () => {
  expect(addressSchema.safeParse(validData).success).toBe(true)
})

test("郵便番号が不正な形式", () => {
  const result = addressSchema.safeParse({ ...validData, postalCode: "1234567" })
  expect(result.success).toBe(false)
  if (!result.success) {
    expect(z.flattenError(result.error).fieldErrors.postalCode).toBeDefined()
  }
})

E2E との役割分担

  • 純粋関数テスト（Vitest / pytest）：ロジックの網羅的な検証。速く、環境依存ゼロ。
  • E2E テスト（Playwright）：「実際にブラウザで操作したときに動くか」の確認。画面遷移・エラー表示・ボタン操作をカバー。

2 層にすることで、「ロジックが壊れているのか」「画面の繋ぎが壊れているのか」を素早く切り分けられます。
